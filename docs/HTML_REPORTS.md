# HTML Reports

RepairGraph generates self-contained, portable HTML reports from workflow
intelligence. Reports are openable locally in any browser — no server, no
framework, no network access required.

All report content is advisory workflow intelligence. It does not certify repair
completion, OEM compliance, or repair quality.

---

## Report structure

### Workflow report

Generated by `build_workflow_html_report(state: RepairState) -> str`.

Sections:

| Section | Contents |
|---|---|
| Header | Session ID, OEM, year, model, operation, status |
| Advisory banner | OEM-primacy advisory note |
| Workflow summary cards | Phase/action/QA/blocker/event counts, next action count |
| Session overview | Full session metadata as key-value pairs |
| Active / blocked phases | Phases currently in-progress or blocked, with zones and pending actions |
| Next recommended actions | Advisory action IDs with type, target, status, and zone refs |
| Open blockers | Blockers by severity and type, with reason and block targets |
| Open QA gates | Open/in-review/failed gates with category, priority, and check text |
| Event timeline | Ordered event ledger as a table |
| Phase overview | All phases with status, zone refs, and action counts |
| Action details | All actions with status, type, target, and zone refs |
| Workflow Timeline Diagram | Mermaid `sequenceDiagram` source |
| Phase Flow Diagram | Mermaid `flowchart LR` source |
| Blocker Flow Diagram | Mermaid `flowchart TD` source |
| Zone Activation Diagram | Mermaid `flowchart LR` source |

### Replay inspector

Generated by `build_replay_html_report(initial_state, events) -> str`.

Sections:

| Section | Contents |
|---|---|
| Header | Session metadata, event count, snapshot count |
| Advisory banner | OEM-primacy advisory note |
| Session overview | Initial session metadata |
| Final state summary | Summary cards for final projected state |
| Replay inspector | Interactive step navigator (vanilla JS) |
| Replay step summary | Table of all steps with event type, actor, target, change count |
| Mermaid sections | Phase flow, blocker flow, workflow timeline (final state) |

---

## Replay UX

The replay inspector is a lightweight vanilla-JS component embedded in the HTML report.
No frameworks, no external libraries.

**Controls:**
- Prev / Next buttons for sequential stepping
- Step chips (one per event) for direct navigation
- Step counter showing current position

**Per-step display:**

Each step shows four cards:

1. **Event** — event ID, type, actor, target (type:ID), timestamp, notes
2. **State After Event** — session status, active phases, completed phase/action counts, open blockers, open QA gates
3. **Changes** — count and human-readable diff descriptions (e.g. `session: not_started → in_progress`)
4. **Next Actions** — advisory next action IDs recommended at this step (when present)

Step data is embedded as JSON in a `<script>` block at page load. No network requests
are made during navigation.

## Mermaid integration

Mermaid diagram sources are embedded in styled `<pre>` code blocks. The reports do
not load or execute the Mermaid JS library. This ensures:

- No CDN dependency
- No external network access
- Deterministic, portable output
- Compatibility with strict offline environments

To render a diagram:
1. Copy the contents of any `mermaid-block` in the report
2. Paste into a Mermaid-compatible tool (Mermaid Live, VS Code extension, etc.)

All four diagram types are included in the workflow report. The replay report
includes phase flow, blocker flow, and workflow timeline for the final projected state.

## Intended use cases

- **Estimator review** — inspect repair state, blockers, and next actions before authorizing work
- **QA audit** — verify which QA gates are open and what they block
- **Technician briefing** — understand the current workflow position and next steps
- **Replay inspection** — step through events to understand how workflow state evolved
- **Operational archival** — portable, self-contained record of a workflow intelligence snapshot

---

## Non-goals

HTML reports are not:
- Production SaaS UIs
- Multi-user dashboards
- Authenticated views
- Live-updating interfaces
- Certified repair records
- OEM procedure replacements

They are portable, local intelligence artifacts — a snapshot of RepairGraph's
workflow projection at a point in time.

## CLI usage

```bash
python -m repairgraph.state.report_cli
```

Outputs:

```
data/extracted/state/accord_workflow_report.html
data/extracted/state/accord_replay_report.html
```

Prints: phase count, open blocker count, event count, replay snapshot count,
output paths, and advisory note.

## API endpoint

```bash
# Default (workflow view)
curl http://localhost:8000/internal/state/accord/report

# Workflow report
curl http://localhost:8000/internal/state/accord/report?view=workflow

# Replay inspector
curl http://localhost:8000/internal/state/accord/report?view=replay
```

Returns `Content-Type: text/html`. No files written. Deterministic output.

---

## Determinism

Both report builders are deterministic: identical `RepairState` input produces
identical HTML output. This property:

- Makes reports reproducible across runs
- Enables diffing report versions as state evolves
- Ensures CLI and API surfaces produce the same output for the same state

The HTML report CLI uses the same `build_accord_projected_state()` builder as
the state CLI, AR payload CLI, and visualization CLI. All tools share the same
deterministic demo foundation from `state/demo.py`.

---

## Advisory

All HTML report content is advisory workflow intelligence derived from RepairGraph
procedure data and explicit state events. Reports are local, portable artifacts —
not certified repair records or OEM-compliance documents. Qualified technician
review and OEM procedure verification are required before acting on any
workflow recommendation.
