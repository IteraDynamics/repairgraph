# RepairGraph Compiler Model

## Core Architectural Insight

RepairGraph increasingly resembles a compiler pipeline.

OEM repair procedures act as the source language.

RepairGraph normalization and ontology layers act as the intermediate representation.

Future graph exports, query systems, operational copilots, and spatial overlays become downstream execution layers.

---

# Conceptual Pipeline

```text
OEM Procedure PDF
    ↓
Text Extraction
    ↓
Semantic Matching
    ↓
Ontology Normalization
    ↓
RepairGraph Intermediate Representation
    ↓
Validation
    ↓
Graph Export
    ↓
Operational Intelligence Layers
```

---

# Why This Matters

The project is not fundamentally a document retrieval system.

The project is a procedural intelligence transformation system.

This distinction is important because:

- retrieval systems operate on documents
- intelligence systems operate on representations

RepairGraph aims to transform collision repair procedures into machine-operable structural and procedural representations.

---

# Emerging Graph Layers

## Structural Topology Graph

Represents:
- panels
- separators
- stiffeners
- rails
- flanges
- adjacency relationships

## Procedural Dependency Graph

Represents:
- replacements
- inspections
- joining methods
- corrosion requirements
- sequencing
- operational dependencies

These graph layers overlap but are not identical.

---

# Strategic Direction

The long-term value of RepairGraph likely emerges from:

- normalized repair topology
- operational relationship modeling
- contextual inference
- graph-based procedural reasoning

rather than static document presentation.
